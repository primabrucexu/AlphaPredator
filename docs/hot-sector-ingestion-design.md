# 热点板块数据解析设计文档

> 说明：本文档负责维护“当日热点板块”相关的数据输入、解析、结构化、沉淀与近 3 个交易日派生规则。整体业务边界见 `docs/business-requirements.md`；AI 如何使用这部分数据见 `docs/ai-stock-selection-design.md`；页面展示方式见 `docs/market-display-design.md`。

## 1. 文档边界

### 1.1 本文档负责
- 输入源定义
- 日期解析规则
- 图片到结构化数据的处理链路
- 结构化字段设计
- 归一化规则
- 日度热点板块聚合口径
- 近 3 个交易日派生逻辑
- 解析质量与回溯要求

### 1.2 本文档不负责
- 热点板块在 AI 中是软约束还是硬过滤
- 首页 / 结果页 / 详情页如何展示热点板块结果
- 任何页面级按钮、卡片、图表布局规则

## 2. 设计目标

“当日热点板块”不是泛指任意板块涨幅榜，而是基于固定格式的每日复盘图片数据，解析并保存相关热点板块 / 热点股票数据，再供上层模块使用。

第一版的目标不是做一套复杂的题材知识图谱，而是先建立一条稳定、可复用、可回溯的数据链路。

## 3. 输入源定义

### 3.1 输入格式定位
- `imgs/` 下现有图片是 **示例数据**。
- 这些图片的作用是定义后续解析时需要遵循的输入格式。
- 后续只要继续提供同格式的每日复盘图片，系统就按该格式解析并保存热点数据。

### 3.2 输入源语义
- 输入源不是通用板块排行榜。
- 输入源是固定格式的每日复盘图片。
- 图片中包含的是当天热点股票及其相关逻辑说明。
- “热点板块”需要通过这类图片中的个股信息归纳和沉淀出来。

### 3.3 日期规则
- 图片文件名默认采用 `MMDD.png`。
- 文件名只定义月日，不单独定义年份。
- 年份需要由导入批次或外部上下文补全，不能依赖文件名本身唯一确定。
- 解析后的正式数据必须保存为唯一 `trade_date`。

## 4. 总体数据链路

推荐按下面的顺序处理：

`每日复盘图片 -> 个股行提取 -> 原始逻辑说明清洗 -> 题材 / 板块归一化 -> 日度热点板块聚合 -> 近 3 个交易日派生特征`

第一版的重点是先把数据链路跑通，而不是在最前面追求极高复杂度的自动归因能力。

## 5. 结构化结果分层

第一版建议至少保存：
- **4 层结构化结果**
- **1 层派生结果**

### 5.1 来源层
用于保存原始输入来源和追溯关系，至少包含：
- `trade_date`
- `source_file`
- `source_type`
- `import_batch`
- `parse_status`
- `parse_notes`

### 5.2 个股层
用于保存当天从图片中提取出来的热点股票事实，至少包含：
- `trade_date`
- `stock_code`
- `stock_name`
- `board_count`
- `limit_up_time`
- `reason_raw`
- `reason_clean`
- `source_file`
- `ocr_confidence`
- `needs_review`

这一层必须保留 `reason_raw`，保证后续可以回看原始逻辑说明。

### 5.3 板块映射层
用于保存个股与题材 / 板块之间的归一化关系，至少包含：
- `trade_date`
- `stock_code`
- `sector_name_canonical`
- `sector_alias_hit`
- `is_primary_sector`
- `mapping_method`
- `mapping_confidence`
- `needs_review`

其中：
- 一只股票允许映射多个题材 / 板块
- 但必须有且仅有一条映射记录满足 `is_primary_sector = true`，用于日度聚合

### 5.4 日度热点板块层
用于保存按天聚合后的热点板块结果，至少包含：
- `trade_date`
- `sector_name_canonical`
- `source_stock_count`
- `max_board_count`
- `representative_stock_codes`
- `representative_stock_names`
- `heat_score`
- `rank_today`
- `aggregate_confidence`

### 5.5 近 3 个交易日派生层
用于保存近 3 个交易日的热点板块派生结果，至少包含：
- `days_present_3d`
- `heat_sum_3d`
- `heat_avg_3d`
- `best_rank_3d`
- `latest_rank`
- `trend_tag`

## 6. 近 3 个交易日派生逻辑

你已经明确“近期”默认按近 3 个交易日理解，因此需要单独保存这层派生结果，而不是在上层调用时临时拼算。

其中 `trend_tag` 第一版推荐采用：
- `new`：近 3 日只在最新日出现
- `persistent`：近 3 日出现至少 2 次，且热度仍然靠前
- `fading`：前两日较热，但最新日明显降温

## 7. 归一化与解析规则

### 7.1 原始信息保留原则
- 解析后的结构化结果不能覆盖原始信息
- 必须保留原始图片来源
- 必须保留原始逻辑说明文本
- 清洗后的字段仅作为结构化使用，不替代原始记录

### 7.2 题材 / 板块归一化原则
- 需要维护一层规范化词表
- 词表至少支持：
  - `canonical_name`
  - `aliases`
  - `status`
  - `notes`
- 同义写法、别名、近义表达，需要归并到同一规范板块名下

### 7.3 多题材处理原则
- 同一只股票可以拥有多个题材标签
- 日度聚合只使用 `is_primary_sector = true` 的那条映射记录
- 其他题材标签作为补充信息保留，供后续解释与扩展使用

### 7.4 低置信度处理原则
- 解析结果存在歧义时，不应强行归并
- 需要显式标记 `needs_review`
- 低置信度数据应允许人工复核或后续规则修正

## 8. 热度聚合口径

第一版不强行固定复杂公式，但日度热点板块聚合至少要综合考虑：
- 当天关联股票数量
- 当天最高板高度
- 代表性股票情况

第一版要求：
- 能稳定给出当日热点板块排序
- 能支撑近 3 个交易日持续性观察
- 不要求一开始就把热度公式设计得过于复杂

## 9. 上层消费接口说明

这条数据链路产出的结果，会被上层模块消费。第一版至少需要稳定提供：
- 日度热点板块结果
- 近 3 个交易日派生结果
- 可追溯到来源图片与原始逻辑说明的回溯能力

至于 AI 如何使用这部分数据、页面如何展示这部分数据，分别由对应 owner 文档维护。

## 10. 当前边界

第一版当前不优先做：
- 盘中实时热点追踪
- 复杂的题材知识图谱
- 完整的板块本体系统
- 完全黑盒式的自动归因
- 在本文档中继续定义 AI 与页面层规则

## 11. 验证要求

后续实现或落地时，至少验证：
1. 同格式输入图片能够稳定解析
2. 解析结果能区分：来源层、个股层、板块映射层、日度热点板块层、近 3 个交易日派生层
3. 任一热点板块结果都能追溯到原始来源图片和原始逻辑说明
4. `is_primary_sector` 命名和语义在全文保持一致
5. 近 3 个交易日结果能够表达“轮动”和“持续性”
