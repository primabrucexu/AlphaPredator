# docs/agent 使用说明

本目录用于存放 AI/编码 agent 产出的工作文档，主要承载阶段内的细粒度拆解与执行方案。

## 目录职责

- 记录阶段内目标拆解、方案设计、风险评估、验收标准。
- 为实现过程提供可追溯依据，但不替代项目级规则文档。
- 与 `docs/docs.md` 配合：`docs/docs.md` 只做导航，本目录保存正文。

## 适用场景

- 需求超出当前任务描述，需要先拆分目标再执行。
- 同一需求存在多种实现路径，需要记录方案取舍。
- 需要沉淀阶段内可复用的执行说明。

## 不适用场景

- 修改项目硬规则或事实定义（应由人维护 `docs/human` / `docs/phase.md`）。
- 记录任务状态（已完成/进行中/未完成）。

## 命名规范

- 文件名格式：`phase-<阶段名称>-<目标名称>.md`
- `<阶段名称>` 建议统一使用：`phase1`、`phase2`、`phase3`、`phase4`
- `<目标名称>` 使用 kebab-case，聚焦单一目标

示例：

- `phase-phase1-market-data-import.md`
- `phase-phase2-hot-sector-parser.md`
- `phase-phase3-pattern-backtest.md`

## 推荐模板

```markdown
# <标题>

## 背景

- 需求来源：
- 阶段上下文：
- 约束条件：

## 目标

- 做什么：
- 不做什么：

## 方案

- 方案 A：
- 方案 B：
- 选择理由：
- 影响范围：

## 风险

- 风险点：
- 缓解措施：

## 验收

- 验收标准 1：
- 验收标准 2：
```

## 维护规则

- 与 `docs/human` 冲突时，以 `docs/human` 为准。
- 与 `docs/phase.md` 冲突时，不直接修改 `docs/phase.md`，改为提 issue。
- 本目录文档只做阶段内细化，不改写阶段定义。
- 重要长期文档应在 `docs/docs.md` 增加导航链接。

## 快速开始

1. 先在 `docs/phase.md` 确认当前需求所属阶段。
2. 按规范创建文件：`phase-<阶段名称>-<目标名称>.md`。
3. 按模板最少填写：背景、目标、方案、风险、验收。
4. 若涉及规则冲突，文档中显式标注并以 `docs/human` 为准。
