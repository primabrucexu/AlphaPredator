# AlphaPredator 首页极简化与股票直达搜索设计（Phase 2.10）

## 1. 背景与目标

基于 `phase.md` 的 `Phase 2.10`，首页将从“市场总览看板”改为“品牌入口 + 股票直达搜索”。

目标：

- 首页仅保留 `AlphaPredator` 品牌展示与单一搜索框。
- 支持两种输入：
  - 股票代码（如 `000001`、`000001.SZ`）
  - 拼音简称（如 `PAYH`）
- 点击搜索后直接跳转到详情页：`/stocks/{stock_code}`。
- 异常场景可感知：空输入、无匹配、多匹配、服务异常。
- 首次进入首页时展示“市场数据是否已完成初始化”状态。
- 首页展示两个时间信息：`股票清单更新时间`、`每日行情更新截止时间`。
- 初始化板块在 CSV 上传成功后不再展示上传控件，仅展示股票清单摘要。

## 2. 非目标

- 不实现复杂筛选器（行业/板块/多条件组合）。
- 不在首页保留市场总览、热点板块、股票列表。
- 不在本阶段实现搜索历史、联想推荐、热词推荐。

## 3. 用户流程

### 3.1 主流程（成功）

1. 用户访问首页 `/`。
2. 页面展示品牌区 + 搜索输入框 + 搜索按钮。
3. 用户输入代码或拼音简称并提交。
4. 前端调用“股票解析 API”。
5. 后端返回唯一 `stock_code`。
6. 前端跳转 `/stocks/{stock_code}`。

### 3.2 异常流程

- 空输入：前端即时提示“请输入股票代码或拼音简称”。
- 无匹配：提示“未找到匹配股票”。
- 多匹配：提示“匹配到多只股票，请输入更完整代码/简称”。
- 服务异常：提示“服务暂不可用，请稍后重试”。

### 3.3 首次访问初始化提示流程

1. 用户首次进入首页 `/`。
2. 前端查询初始化概览状态（Token、股票清单、初始化状态、更新时间）。
3. 页面显示“已初始化 / 未初始化”提示。
4. 若未初始化，展示快捷入口跳转初始化页。
5. 若已上传股票清单，初始化区域只展示摘要信息，不显示上传控件。

## 4. 前端设计

## 4.1 页面结构

目标页面：`frontend/src/pages/MarketOverviewPage.tsx`（或新建 `HomeSearchPage.tsx` 并在路由替换）。

结构：

- 品牌区
  - Logo（可先用文字标识）
  - 文案：`AlphaPredator`
- 搜索区
  - 输入框：placeholder 为“输入股票代码或拼音简称”
  - 搜索按钮
- 初始化状态区
  - 初始化完成状态（首次进入即可看到）
  - 股票清单更新时间（`stock_list_updated_at`）
  - 每日行情更新截止时间（`daily_quote_cutoff_time`）
  - 初始化入口按钮（仅未就绪时强调）
- 股票清单摘要区（仅 CSV 已上传后显示）
  - 主板股票数量
  - 创业板股票数量
  - 科创板股票数量
  - 股票清单最后更新时间
  - 该区域显示时，不再展示上传控件
- 结果提示区
  - 仅显示错误/状态提示，不展示列表

## 4.2 交互规则

- 输入提交方式：点击按钮 + 回车提交。
- 输入归一：`trim` + 大写化（便于拼音匹配）。
- 请求中：按钮 loading，防止重复提交。
- 提交成功：立即跳转详情页。
- 首次访问必须触发初始化状态检查并展示状态结果。
- 若股票清单已上传：隐藏上传控件，仅显示摘要信息。
- 若股票清单未上传：显示“去初始化”入口，不阻塞搜索框展示。

## 4.3 路由与导航

- 根路由 `/` 指向极简首页。
- 详情页沿用现有 `/stocks/:stockCode`。
- 侧边菜单可保留，但首页内容改为极简入口。

## 5. 后端设计

## 5.1 新增解析接口

建议接口：

- `GET /api/market/resolve?q={input}`

## 5.2 初始化概览接口（首页展示用）

建议接口：

- `GET /api/data-init/overview`

响应示例：

```json
{
  "init_completed": true,
  "token_configured": true,
  "stock_list_uploaded": true,
  "stock_list_updated_at": "2026-05-03T08:21:00Z",
  "daily_quote_cutoff_time": "2026-05-03T15:30:00+08:00",
  "board_counts": {
    "主板": 3200,
    "创业板": 1300,
    "科创板": 560
  }
}
```

响应示例：

```json
{
  "status": "ok",
  "stock_code": "000001",
  "stock_name": "平安银行",
  "match_type": "code"
}
```

无匹配：

```json
{
  "status": "not_found",
  "message": "未找到匹配股票"
}
```

多匹配：

```json
{
  "status": "ambiguous",
  "message": "匹配到多只股票",
  "candidates": [
    {"stock_code": "000001", "stock_name": "平安银行"},
    {"stock_code": "000002", "stock_name": "万科A"}
  ]
}
```

## 5.3 匹配优先级

1. 股票代码精确匹配（`000001` / `000001.SZ`）。
2. 拼音简称精确匹配（来自 CSV 的 `cnspell`）。
3. 拼音简称前缀匹配（用于兼容输入不完整）。

## 5.4 服务层职责

在 `MarketDataService` 增加解析方法：

- 输入：`query: str`
- 输出：`status + stock_code/candidates`
- 不负责跳转，仅返回解析结果。
- 提供首页初始化概览查询（初始化状态、股票清单摘要、更新时间字段）。

## 6. 数据依赖与模型

## 6.1 数据来源

- 股票清单来自 CSV 上传（Phase 2.9 已定义）。
- CSV 字段包含：`ts_code,symbol,name,industry,cnspell,market,list_date,list_status,delist_date`。

## 6.2 数据落库要求

为支持拼音搜索，数据库中需要可查询字段：

- `stock_code`（代码）
- `stock_name`（名称）
- `cnspell`（拼音简称）

可选落地策略：

- 方案 A：在现有 `stock_profiles` 增加 `cnspell` 字段（推荐，改动小）。
- 方案 B：新增 `stock_universe` 表并由服务层 join 查询。

## 6.3 索引建议

- `stock_profiles(stock_code)` 主键已具备。
- 新增 `cnspell` 索引提升拼音解析性能。

## 7. 接口契约变更

保持原有市场接口不变，新增：

- `GET /api/market/resolve`
- `GET /api/data-init/overview`

前端 `api.ts` 新增：

- `resolveStockInput(query: string)`
- `getInitOverview()`

## 8. 错误处理矩阵

- 输入空：`400`（或前端拦截，不发请求）
- 参数非法：`400`
- 无匹配：`200 + status=not_found`
- 多匹配：`200 + status=ambiguous`
- 服务失败：`500`

## 9. 验收标准

满足以下条件视为 Phase 2.10 设计落地完成：

1. 首页仅包含品牌与单一搜索框。
2. 输入股票代码可直接跳转详情页。
3. 输入拼音简称可解析并跳转详情页。
4. 空输入/无匹配/多匹配/服务异常均有明确提示。
5. 不依赖首页市场总览数据也可完成直达搜索。
6. 首次进入首页时可看到“是否完成初始化”状态提示。
7. CSV 上传成功后，初始化模块不再显示上传控件，仅显示板块数量与更新时间摘要。
8. 首页可展示股票清单更新时间和每日行情更新截止时间。

## 10. 实施拆分（建议）

- A 阶段：前端极简首页布局改造。
- B 阶段：后端解析接口与服务层能力。
- C 阶段：CSV 字段落库对齐（含 `cnspell`）与索引。
- D 阶段：联调与回归（后端测试 + 前端构建）。

## 11. 风险与对策

- 拼音字段缺失：上传 CSV 时强校验并给出缺失提示。
- 多匹配过多：先返回 ambiguous，后续可加候选弹窗。
- 数据未初始化：接口返回“股票清单未准备完成”。
- 性能问题：为 `cnspell` 建索引，避免全表扫描。
- 更新时间口径不一致：统一由后端返回 `stock_list_updated_at` 与 `daily_quote_cutoff_time`。
