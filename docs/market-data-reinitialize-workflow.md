# 市场数据重新初始化设计文档

> 说明：本文档负责维护“重新初始化市场数据（reinitialize）”能力，用于在新增字段、补充派生逻辑或升级数据结构后，基于**已有股票列表**重新构建市场数据。涨跌停字段如何计算见 `docs/price-limit-rule-design.md`；详情页如何消费字段见 `docs/stock-detail-page-design.md`；K 线如何渲染涨跌停语义见 `docs/kline-limit-color-design.md`。

## 1. 文档边界

### 1.1 本文档负责
- 重新初始化的业务定义
- 与“全量初始化”“数据重置”的区别
- 保留数据与重建数据的边界
- 前后端入口建议
- 状态管理与幂等性要求
- 字段新增后的推荐处理路径

### 1.2 本文档不负责
- 单个字段的具体计算公式
- K 线页面的颜色、图表和交互细节
- 热点板块等独立数据链路的重建策略
- AI 侧如何使用重建后的新字段

## 2. 背景与目标

随着 AlphaPredator 持续演进，市场数据模型会新增字段，例如：

- `is_st`
- `limit_up_price`
- `limit_down_price`
- `is_limit_up`
- `is_limit_down`
- 新的技术指标或详情页展示字段

这些字段加入后，用户不应被要求再次上传股票列表 CSV。系统需要一条独立能力，能够在保留当前股票池和配置的前提下，清空并重建派生市场数据。

目标：

1. 用户无需重新上传股票列表，即可重建市场数据。
2. Token 与股票池元数据默认保留。
3. 新字段可以通过重初始化补齐到现有数据链路。
4. 重初始化可重复执行，且不会产生重复或脏数据。

## 3. 概念区分

### 3.1 全量初始化
- 面向首次导入场景
- 要求已配置 Token
- 通常要求用户先上传股票列表
- 以“从零开始构建市场数据”为目标

### 3.2 数据重置
- 面向“回到初始状态”场景
- 可能清空数据库、parquet、状态文件、股票列表、Token
- 目标是清理所有本地数据

### 3.3 重新初始化
- 面向“模型演进后的重建”场景
- 保留股票列表与 Token
- 只清理并重建依赖股票池生成的市场数据
- 不要求重新上传 CSV

## 4. 重新初始化的定义

“重新初始化市场数据”定义为：

> 基于当前已保存的股票池与配置，重新执行一次市场数据初始化流程，用于新增字段补算、派生逻辑修复、结构升级后的数据重建。

其本质是：

- **保留基础配置**
- **删除旧的派生市场数据**
- **按当前代码与规则重新构建**

## 5. 保留与重建范围

## 5.1 默认保留的数据

- Tushare Token
- `stock_universe`
- 原始股票列表文件（若项目保留）
- 股票列表更新时间与相关摘要元数据

## 5.2 需要重建的数据

- `daily_stock_snapshots`
- `stock_profiles`
- `daily_bars`
- `market_snapshot.json`
- `parquet` 中依赖市场初始化生成的行情文件
- `data/imports/market-data/` 下的市场数据批次文件
- 初始化状态结果

## 5.3 默认不动的数据

- 与股票池无直接依赖的独立模块数据
- 热点板块图片原始输入
- 独立导入链路产生的其他业务数据

## 6. 适用场景

以下场景推荐触发重新初始化，而不是要求用户重新上传 CSV：

1. 新增日行情或详情页字段
2. 新增涨跌停派生字段
3. 新增 `is_st`、标签、摘要或技术指标字段
4. 修复初始化过程中的结构化逻辑 bug
5. 修改批次写入结构，需要重建旧数据

## 7. 与涨跌停链路的关系

对于 `docs/price-limit-rule-design.md` 中定义的涨跌停字段，重新初始化是推荐补齐方式。

当新增以下字段时：

- `limit_up_price`
- `limit_down_price`
- `limit_pct`
- `is_limit_up`
- `is_limit_down`
- `limit_rule`
- `limit_status`
- `limit_rule_version`

推荐处理路径为：

`已有 stock_universe -> 重新初始化 -> 重新拉取/重建日行情 -> 计算涨跌停字段 -> 批次入库 -> 页面消费`

这条链路保证：

- 前端不做临时推断；
- 详情页与 K 线颜色渲染读取的是已固化字段；
- 新字段可在已有股票池基础上补齐。

## 8. 前后端入口建议

## 8.1 后端接口

建议新增独立接口：

- `POST /api/data-init/reinitialize`

语义：
- 不重新上传股票列表
- 不清空 Token
- 基于现有股票池重建市场数据

## 8.2 前端入口

建议在初始化页面提供单独按钮：

- `重新初始化市场数据`

并在确认弹窗中明确提示：

- 将保留当前股票列表与 Token 配置
- 将清空并重建市场行情、日线、快照与派生字段
- 适用于新增字段或规则升级后的数据重建

## 9. 运行状态建议

为了区分首次初始化与重初始化，建议初始化状态增加：

- `operation: init | reinitialize`
- 可选 `phase: loading_universe | fetching_bars | building_batches | importing | done`

要求：
- 当存在 `running` 任务时，不允许再次触发重初始化；
- 前端能区分“首次初始化中”与“重新初始化中”。

## 10. 幂等性要求

重初始化必须支持重复执行。

要求：

1. 重复触发不产生重复数据
2. 失败后允许再次执行
3. 成功后再次执行结果仍然一致
4. 中途失败不会留下无法恢复的脏状态

推荐策略：

- 先清理目标派生表/文件
- 再完整重建
- 使用明确的重建目标集合，不做模糊删除
- 初始化与重初始化共用一套核心构建逻辑

## 11. 决策表：何时增量更新，何时重初始化

| 场景 | 推荐动作 |
|---|---|
| 每日新增一个交易日行情 | 每日更新 |
| 新增详情页展示字段，但可由现有数据实时推导 | 视情况增量补算 |
| 新增需要历史回填的日度字段 | 重新初始化 |
| 新增涨跌停固化字段 | 重新初始化 |
| 修复历史计算 bug | 重新初始化或专项回填 |
| 想彻底清空环境回到初始状态 | 数据重置 |

## 12. 验收要求

满足以下条件视为该链路设计完成：

1. 用户无需重新上传 CSV，即可重建市场数据。
2. Token 与 `stock_universe` 在重初始化后仍保留。
3. 新增字段可通过重初始化补齐到历史数据链路。
4. 重初始化不会产生重复行或脏状态。
5. 涨跌停等派生字段可沿“计算 -> 落库 -> 页面消费”完整贯通。

## 13. 文档链路职责

- `docs/price-limit-rule-design.md`：负责“涨跌停字段怎么算、何时算、如何固化”。
- `docs/market-data-reinitialize-workflow.md`：负责“新增字段后，何时需要重建、如何重建”。
- `docs/stock-detail-page-design.md`：负责“详情页需要消费哪些字段”。
- `docs/kline-limit-color-design.md`：负责“详情页中的 K 线如何表达涨跌停视觉语义”。

四者合起来构成完整链路：

`规则定义 -> 初始化/重初始化 -> 接口输出 -> 页面渲染`
